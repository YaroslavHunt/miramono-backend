# План реалізації бекенду MIRAMONO

Бекенд для сайту приватної стоматологічної практики MIRAMONO Dentistry. Стек: Node 24, NestJS 11,
TypeScript, PostgreSQL + TypeORM 1.0, Redis (кеш), Docker. Реалізація поділена на послідовні фази.

## Архітектура

- `src/config/` — типізована конфігурація + валідація `.env`.
- `src/database/` — `DataSource`, snake-стратегія, міграції.
- `src/common/` — базова сутність, фільтри, інтерсептори, гуарди, декоратори.
- `src/modules/<feature>/` — фічеві модулі (controller, service, dto, entities, тести).
- `src/health/` — health-check.
- Публічні ендпойнти на читання + захищені адмінські CRUD (RBAC).

## Принципи

- Просто, але професійно; швидко; без зайвого ускладнення.
- Кожна фаза завершується повним проходженням перевірок: типізація, лінт, збірка, тести та реальний запуск.
- `.env` завжди повністю заповнений; `.env.example` синхронізований.

## Фаза 0 — Фундамент ✅ (виконано)

Каркас застосунку: NestJS 11 + TypeORM 1.0 + PostgreSQL + Redis-кеш, конфіг із валідацією env,
глобальні pipe/filter, throttling, helmet/CORS, Swagger (`/docs`), health-check (`/health`),
Docker + docker-compose, скрипти міграцій, юніт- та e2e-тести.

## Фаза 1 — Авторизація та користувачі ✅ (виконано)

- Сутність `User` (роль: PATIENT / DOCTOR / ADMIN; вхід за `login` + пароль, контактні дані).
- Реєстрація пацієнта, вхід, JWT access+refresh (ротація, хеш refresh у БД), вихід.
- Хешування паролів argon2; `RolesGuard` + `@Roles()`; декоратор `@CurrentUser()`; `@Public()`.
- Профіль пацієнта (особистий кабінет: перегляд/редагування своїх даних).
- Перша міграція. Сид адміністратора (за потреби — через скрипт/змінні env).

## Фаза 2 — Каталог послуг, лікарі, інформація про клініку ✅ (виконано)

- `ServiceCategory` (напрями лікування) + `Service` (назва, опис, ціна, тривалість, категорія).
- `Doctor` (ім'я, спеціалізація, опис, фото, активність, порядок сортування).
- `ClinicInfo` / контакти (телефон, адреса, графік, соцмережі, статистика «10+/3000+/15+»).
- Публічні ендпойнти на читання (з пагінацією/фільтрами) + адмінські CRUD.

## Фаза 3 — Онлайн-запис і швидка консультація ✅ (виконано)

- `Appointment` (пацієнт або гість, послуга/лікар, бажаний час, статус, коментар, контакти).
- Створення запису без обов'язкової реєстрації (збір контактів) + запис із кабінету.
- Проста доступність лікаря (робочі години/зайняті слоти) без важкої логіки розкладу.
- Адмінське керування заявками (перегляд, зміна статусу) — ядро CRM.
- Форма швидкої консультації (`ConsultationRequest`).
- Спільний `OptionalJwtAuthGuard` для ендпойнтів, що працюють і з токеном, і без нього.

## Фаза 4 — Контент залученості ✅ (виконано)

- `Article` (блог: slug, контент, обкладинка, статус публікації, дата).
- `Review` (відгуки пацієнтів: автор, текст, рейтинг, модерація/публікація).
- `GalleryCase` (галерея до/після: фото before/after, опис, лікар/послуга).
- Завантаження зображень (локальне сховище `/uploads`, валідація типу/розміру).

## Фаза 5 — Платежі, мультимовність, SEO, кеш, фінал ✅ (виконано)

- Платежі: інтеграція LiqPay (sandbox) — створення платежу + серверний callback із перевіркою підпису.
- Мультимовність: `nestjs-i18n` для повідомлень; JSONB-переклади (`uk`/`en`) для статей + резолвер локалі.
- SEO: динамічний `sitemap.xml`, `robots.txt`, мета-поля на статтях.
- Кешування важких публічних відповідей через Redis з інвалідацією (версіоновані неймспейси).
- Фінал: сид демо-даних, повна перевірка, оновлення README та документації API.

**Бекенд завершено** — усі 6 фаз (0–5) виконані. Можливі дальші кроки: e-mail/SMS-нотифікації
про записи й платежі, CI/CD та деплой, моніторинг/логування (OpenTelemetry), повна локалізація всього
контенту (не лише статей), вебхуки/інтеграція з реальним кабінетом LiqPay.
